CTF Write-ups

Sanitized methodology summaries — no flags, no live-target details.

WEB User Profile Access Control ＋

Objective: Determine whether user profile information was properly protected and identify any unauthorized access issues.

Approach: Reviewed the application's authentication flow and examined how profile pages were accessed after logging in. Observed how different user accounts interacted with the same feature and compared responses while staying within the challenge environment. Focused on identifying inconsistencies in authorization rather than attempting to exploit real systems.

Outcome: Learned how Broken Access Control (IDOR - Insecure Direct Object Reference) can occur when applications verify authentication but fail to enforce proper authorization. The challenge reinforced the importance of server-side permission checks.

CRYPTO Base64 Encoding Challenge ＋

Objective: Determine the original message from an encoded string.

Approach: Recognized the format as a common text encoding rather than encryption. Used standard decoding techniques to recover the original content and verified the output for consistency.

Outcome: Improved understanding of the difference between encoding, encryption, and hashing. Learned that Base64 is designed for data representation and does not provide confidentiality.

FORENSICS Network Capture Investigation ＋

Objective: Analyze a packet capture (PCAP) file to identify significant network activity.

Approach: Reviewed captured network traffic, identified relevant protocols, and examined communication patterns within the challenge dataset. Focused on understanding the sequence of events rather than performing active exploitation.

Outcome: Developed skills in network traffic analysis and learned how packet captures can help reconstruct events during an investigation. The challenge reinforced the value of protocol analysis and systematic evidence review in digital forensics.

Technical Notes

Notes on how this site itself was built securely.

Static by design

No database or server-side app behind this site means no SQL injection surface, no session handling, and a much smaller attack surface overall.

Client-side input handling

The contact form below validates field length, format, and required fields before submission, and never renders user input as raw HTML.

What production would add

A real deployment would route the form through a serverless function that re-validates and sanitizes all input server-side, applies rate limiting, and never trusts the client.

Dependencies kept minimal

One external font request, zero JS frameworks, zero tracking scripts — fewer third-party scripts to audit or worry about.
